## Overview

This change adds a learning layer above the existing self-evolve replay and
gate loop.

The existing closed loop remains:

1. build or load baseline trajectory evidence
2. infer a mutable target
3. generate a candidate
4. replay with candidate overlay
5. evaluate baseline/candidate trajectories
6. run gates
7. apply and verify when policy allows

The new layer changes how candidates are produced:

1. collect a trajectory set for the target and task family
2. extract normalized lessons from failures and successes
3. generate a population of local candidate patches
4. merge or select candidates with lineage memory
5. materialize replayable candidate skill content
6. normalize accepted skill content for runtime release

The goal is to make improvements transferable and stable, especially when a
single trajectory already has a high baseline score and only small verified
deltas are acceptable.

## Architecture

### Trajectory Set

A trajectory set is a framework-owned input bundle containing one or more
trajectory records plus optional evaluator reports and replay diagnostics.

Supported members:

- `baseline`: original trajectory from the task or dataset
- `candidate_replay`: replay trajectories generated by candidate overlays
- `accepted_followup`: later production trajectories that ran with accepted
  skills active
- `rejected_candidate`: trajectories and reports from previously rejected
  candidates
- `operator_added`: explicit additional trajectories supplied by the user or
  scheduler

Each member MUST record:

- trajectory id or file path
- task input digest
- target id and target type, when known
- candidate id, when applicable
- source run id, when applicable
- score/evaluator summaries, when available
- evidence quality metrics, when available
- replay confidence and failure diagnostics, when available

The trajectory set is not an unbounded log dump. It should use bounded
excerpts, evidence bundle references, and evaluator summaries. Raw artifacts
remain addressable by path through existing read-only artifact mechanisms.

The explicit file form MUST use a versioned JSON contract. The initial contract
is `aworld.self_evolve.trajectory_set.v1`:

```json
{
  "schema_version": "aworld.self_evolve.trajectory_set.v1",
  "set_id": "podcast-grounding-2026-07",
  "target": {
    "target_type": "skill",
    "target_id": "web-content-grounding"
  },
  "members": [
    {
      "member_id": "baseline-1",
      "role": "baseline",
      "trajectory_path": "trajectories/baseline.log",
      "task_id": "task_20260708105454",
      "task_input_digest": "sha256:...",
      "source_run_id": null,
      "candidate_id": null,
      "evidence_bundle_path": "optional/evidence_bundle.json",
      "evaluator_report_path": "optional/report.json",
      "metadata": {}
    }
  ]
}
```

Member roles are limited to `baseline`, `candidate_replay`,
`accepted_followup`, `rejected_candidate`, and `operator_added`. Relative paths
resolve from the trajectory-set file directory. Absolute paths are allowed only
inside trusted workspace roots or `.aworld/self_evolve` artifact roots.

The loader should reject or deterministically merge duplicate members using
`task_input_digest`, `role`, `candidate_id`, and `source_run_id`. Cross-target
members are not eligible for direct patch generation by default; they may only
contribute advisory lessons unless explicitly admitted by policy. The default
member limit should be bounded, for example 100 members, with diagnostics that
identify the member index, field name, reason, and repair hint.

### Lesson Extraction

Lesson extraction converts trajectory-set members into normalized records.

Required record types:

- `failure_cause`: causal reason a run failed or underperformed
- `failure_memory`: reusable behavior that would prevent a repeated failure
- `success_memory`: reusable behavior that should be preserved
- `lean_solution_path`: minimal successful execution path with failed attempts
  removed
- `required_runtime_behavior`: concrete behavior that future candidates must
  exhibit or preserve
- `harness_diagnostic`: evidence-backed diagnosis of context, workflow, tool,
  evaluation, memory, permission-boundary, or artifact-lifecycle problems that
  affected the run but should not be copied directly into runtime skill content

Each lesson record MUST include:

- `lesson_id`
- `lesson_type`
- `title`
- `summary`
- `evidence_refs`
- `target_scope`
- `generality`
- `confidence`
- `source_run_ids`
- `source_task_ids`
- optional `metrics`

The extractor should prefer normalized metrics and bounded evidence over raw
tool transcripts. For example, evidence issues such as compacted output,
missing evidence bundles, replay argument compaction, or low B2 efficiency
should become concise failure memories rather than large prompt fragments.

### Evidence Trust Boundary

Trajectory evidence is untrusted input. Tool outputs, page text, transcripts,
artifact contents, and evaluator excerpts may contain secrets, private paths,
PII, or prompt-injection text. Lesson extraction and candidate generation should
therefore use a minimization pipeline:

1. classify evidence as raw, bounded summary, digest, artifact ref, or
   normalized metric
2. redact secret-like tokens, authorization headers, cookies, private local
   paths, emails, phone numbers, and other configured sensitive patterns
3. preserve source references through evidence ids, artifact ids, and digests
4. quote prompt-injection-like strings as evidence, never as instructions
5. write released skills from runtime-safe behavior constraints rather than raw
   evidence text

Reports may include enough diagnostics to explain decisions, but they should not
leak sensitive raw text. Optimizer prompts should prefer normalized fields such
as `evidence_compacted`, `evidence_incomplete`, `artifact_backed`, and
`required_behavior` over raw transcripts.

### Harness Diagnostics As Advisory Lesson Memory

Harness diagnostics record problems in the system around the target skill. They
answer a different question from runtime behavior lessons: not only "what should
the skill do differently?", but also "did the surrounding harness provide the
right context, workflow, tools, evidence, memory, permissions, and evaluation?"

A `harness_diagnostic` lesson MUST include the common lesson fields plus:

- `harness_diagnostic_kind`: one of `context_failure`, `workflow_failure`,
  `tool_protocol_failure`, `evaluation_failure`, `memory_failure`,
  `permission_boundary_failure`, or `artifact_lifecycle_failure`
- `severity`
- `reproducibility`
- `suggested_prevention`
- `promotion_status`, defaulting to `advisory`

Initial extraction can be rule-backed. For example:

- compacted, truncated, or missing evidence can become `context_failure`
- missing evidence files or stale artifact paths can become
  `artifact_lifecycle_failure`
- replay argument compaction or unstable replay setup can become
  `workflow_failure`
- incorrect tool ordering or missing required reads can become
  `tool_protocol_failure`
- score gains paired with groundedness, compliance, or held-out regressions can
  become `evaluation_failure`
- repeated rejected variants can become `memory_failure`
- protected-path mutation attempts can become `permission_boundary_failure`

Harness diagnostics are advisory memory by default. Candidate generation may use
them as strategy hints, but production skills must only receive runtime-safe
behavior deltas that are evidence-backed, replayed, gated, and release
normalized. This change records harness diagnostics for visibility and future
guidance; it does not allow candidates to mutate `aworld.self_evolve` framework
code or other harness implementation.

This capability should reuse existing AWorld self-evolve diagnostic sources
rather than creating a parallel runtime telemetry system:

- `EvaluationSummary.metrics` remains the primary normalized metric carrier for
  evaluator, replay, and candidate comparison signals.
- `GateResult` remains the primary source for failed policy, verification,
  budget, confidence, evidence-quality, and protected-path decisions.
- `normalize_feedback_summary()` remains the compact optimizer-facing view of
  metrics, evidence issues, failed gates, required behaviors, and repair plans.
- `TracePack` and `TraceEvidenceStep` remain the source of evidence ids, tool
  names, omitted-step counts, and compression summaries.
- `FilesystemSelfEvolveStore` remains the artifact writer under
  `.aworld/self_evolve/<run_id>/`.

The new work is to promote those existing signals into first-class
self-evolve-owned records. The implementation should add a small
`HarnessDiagnostic` contract and extractor, preferably under
`aworld.self_evolve`:

- `HarnessDiagnosticKind`
- `LessonPromotionStatus`
- `HarnessDiagnostic`
- `extract_harness_diagnostics(...)`
- `write_harness_diagnostics(...)`

The extractor should accept already-normalized inputs such as evaluation
summaries, gate results, feedback summaries, trace packs, replay diagnostics,
and prior-run lineage. It should not require callers outside
`aworld.self_evolve` to emit harness-specific records. This keeps the feature an
extension of self-evolve learning, not a new global AWorld runtime contract.

### Success-Preserving Optimization

When the baseline or previous accepted candidate is already high scoring,
candidate generation MUST preserve the observed winning path unless a lesson
shows a concrete defect. This mode is the default for high-baseline cases.

High-baseline generation should:

- preserve success memories and lean solution paths
- avoid adding broad validation loops
- avoid adding new external claims or unrelated workflows
- prefer small improvements in evidence durability, step count, or robustness
- allow a no-op recommendation when no lesson-backed safe delta exists

The runner should record when a no-op is selected because the evidence does not
justify a mutation. A no-op result is a valid optimization outcome and should
not be reported as a framework failure.

### Candidate Population

Candidate generation should produce a small population of candidate patch
strategies before materializing full skill content.

Each candidate strategy should include:

- candidate family name
- intended behavior delta
- lessons addressed
- harness diagnostics considered, when any influenced the strategy
- success behaviors preserved
- expected metric impact
- risk notes
- replay priority
- patch operations or semantic patch intent

The default population size should be small and budget-aware. The framework
should replay only the top-ranked candidates by default. Additional candidates
remain persisted for audit and later runs.

The initial defaults should be conservative:

- generated population size: 5
- max replayed candidates: 2
- hard max replayed candidates: 5

Candidate replay priority should be deterministic. A recommended default score
is:

```text
priority_score =
  lesson_coverage * 0.30
+ preserves_success_path * 0.25
+ expected_metric_gain * 0.20
+ evidence_confidence * 0.15
- risk * 0.10
- complexity_penalty * 0.10
```

Tie breakers should be deterministic:

1. fewer patch operations
2. smaller runtime instruction diff
3. higher severity lesson coverage
4. candidate id lexical order

If all candidates fall below the no-op threshold, the runner should produce a
no-op optimize result rather than replaying weak candidates. If replay budget is
exhausted, non-replayed candidates remain in the population artifact with
`not_replayed_due_to_budget`.

### Patch-Oriented Candidate Materialization

Candidates should be generated as local edits first, then materialized into a
full `SKILL.md` only after validation.

Supported edit intents should include:

- append to section
- replace text in section
- add section
- create reference file
- delete obsolete section
- normalize release text

The materializer must validate:

- front matter remains valid
- skill id and description remain valid
- release metadata remains separate from runtime instructions
- reference files are linked from `SKILL.md`
- no protected files are modified
- resulting skill size stays within policy

This design keeps generated changes reviewable and prevents whole-file rewrites
from drifting away from proven successful behavior.

### Lineage And Lesson Memory

Lineage memory persists what was tried and why.

It should track:

- lesson ids used to generate each candidate
- parent candidate ids
- candidate content fingerprint
- semantic fingerprint of behavior delta
- replay/evaluator outcomes
- accepted/rejected status
- rejection gates
- release-normalized content fingerprint

Duplicate detection should use:

- exact content fingerprint
- normalized semantic fingerprint
- lesson-set fingerprint

This prevents repeated rejected variants while still allowing a new candidate
to reuse an old lesson with a materially different behavior delta.

### Release Normalization

Candidate content may contain self-evolve internal context during proposal and
evaluation. Production skills should not.

Before verified apply writes or activates a skill, release normalization MUST
convert the candidate into runtime-facing instructions:

- remove source task ids from the instruction body
- remove baseline/candidate score language
- remove evaluator rubric names such as A1/B2 unless they are in metadata
- remove "candidate_score exceeds baseline_score" style acceptance checks
- preserve runtime behavior constraints that passed gates
- preserve release metadata in front matter or sidecar metadata

For example, a candidate section like:

```text
candidate_score exceeds baseline_score; preserve A1_groundedness; source task
ids: task_...
```

should become runtime guidance such as:

```text
When answering from external content, first capture bounded evidence, then
answer only from claims present in that evidence. If evidence is compacted,
retry with a narrower structured extraction before finalizing.
```

The post-apply loader verifier should verify the normalized released content,
not the pre-normalization internal candidate content.

Post-apply loader verification is necessary but not sufficient. Because
normalization can rewrite candidate instructions after replay and gates, the
runner must verify the normalized content before apply. It can do so by either:

- semantic equivalence: confirm every gate-critical runtime behavior constraint
  from the accepted candidate is preserved in normalized content
- replay fallback: replay and gate the normalized candidate content when
  equivalence cannot be established

The normalization result should carry:

- pre-normalization fingerprint
- normalized release fingerprint
- preserved runtime constraints
- mapping from lesson ids or gate-critical behaviors to normalized text refs
- normalization verification status

If normalization removes a behavior needed for accepted gates, apply must be
rejected before writing the formal skill.

### Reports

Run reports should add a trajectory-set section:

- trajectory set id
- included trajectory members
- extracted lesson counts by type
- top failure memories
- top success memories
- harness diagnostics by kind, including advisory diagnostics not promoted into
  candidate generation
- chosen candidate population
- selected candidate lineage
- release normalization summary

Reports should make clear whether acceptance was based on:

- single trajectory
- repeated single-case replay
- trajectory set validation
- held-out dataset validation

### CLI Surface

`aworld-cli optimize` may add thin inputs:

- `--from-trajectory-set <path>`
- `--include-prior-runs`
- `--trajectory-set-name <name>`

The CLI must not implement lesson extraction, candidate population, lineage
memory, replay ranking, gates, or release normalization. It only parses
arguments, displays progress, and calls framework APIs.

Single trajectory usage remains valid:

```bash
aworld-cli optimize --from-trajectory path/to/trajectory.log --apply auto_verified
```

Trajectory set usage should be additive:

```bash
aworld-cli optimize --from-trajectory-set path/to/set.json --apply auto_verified
```

### Built-In Self-Evolve Skill Boundary

The built-in `self_evolve` skill may describe how to operate trajectory-set
optimize flows, but framework capabilities stay in `aworld.self_evolve`.

The skill may include:

- how to collect related trajectories
- how to inspect run reports
- how to decide whether to run single-trajectory or trajectory-set optimize
- warnings not to manually edit target skills

The skill MUST NOT contain implementation-specific target inference logic,
gate bypass logic, replay orchestration, or release write behavior.

## Artifact Layout

Each trajectory-set optimize run should persist:

```text
.aworld/self_evolve/<run_id>/
  report.json
  target_selection.json
  trajectory_set/
    set.json
    members/
      <member_id>.json
  lessons/
    lessons.jsonl
    harness_diagnostics.jsonl
    extraction_report.json
  population/
    candidates.jsonl
    patches/
      <candidate_id>.json
  candidates/
    <candidate_id>/SKILL.md
    <candidate_id>/release_normalized/SKILL.md
  replay/
    <candidate_id>/...
  evaluation/
    <candidate_id>/...
  apply/
    <candidate_id>.backup.md
    <candidate_id>.journal.json
```

Existing artifact paths may remain, but reports must link these sections when
present.

## Validation Strategy

Unit tests should cover:

- trajectory set parsing and member validation
- lesson extraction schema normalization
- harness diagnostic extraction and advisory promotion status
- success/failure memory compaction
- high-baseline no-op or conservative-delta behavior
- candidate population ranking
- lineage duplicate detection
- release normalization removing internal self-evolve language
- post-apply verification using normalized content

Integration tests should cover:

- optimizing from a single trajectory remains compatible
- optimizing from a small trajectory set creates lesson records and candidate
  population artifacts
- harness diagnostics are reported without being copied into released skill
  instructions
- rejected candidates influence later candidates without permanently blocking
  a new materially different candidate
- accepted candidates are release-normalized before runtime visibility

## Migration Notes

Existing `.aworld/self_evolve` reports remain valid. Runs without trajectory-set
metadata should be treated as single-trajectory runs. Prior rejected and
accepted candidates can be imported into lineage memory lazily when a target is
optimized again.
