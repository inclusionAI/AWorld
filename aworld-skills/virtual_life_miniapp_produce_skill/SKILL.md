# 虚拟人生/人生副本类灵光小程序制作指南 (Virtual Life Mini-App Production Skill)

本指南旨在为 AI 智能体提供一套标准化的工作流，用于从零开始构建“虚拟人生”、“人生副本”或“互动视觉小说”类的灵光小程序。请严格按照以下流程和规范执行。

## 第一阶段：背景调研与剧情设计 (Story Design)

本阶段的核心目标是指导策划与AI在设计剧情时，如何规范、细致地输出分页剧情推进文档（`page_design.md` 或 `page_design.txt`）。根据用户需求，剧情设计分为**两种模式**：

### 模式一：标准收束模式 (Standard Convergent Mode / 幻觉选择)
- **核心理念**：剧情包含 N 个关键的分叉点。无论用户在分叉点选择A还是B，剧情在给出不同的即时反馈后，都必须巧妙地**收束到同一条主线上**，最终走向同一个具有警示意义的结局。
- **适用场景**：强调宿命感、不可逆的悲剧（如网贷、赌博等），开发工作量较小。

### 模式二：高复杂度发散模式 (High Complexity Divergent Mode / 网状多结局)
- **核心理念**：每一个选择的分叉之后，确实会导致走向不同的剧情。例如5个分叉点，理论上最多可达 2^5 = 32 个结局。为了剧情合理性，前期的某些分叉可以局部收束，但中后期必须发散出**多个完全不同的结局**（如 10-20 个结局）。
- **适用场景**：强调“蝴蝶效应”，给用户真正的沉浸式体验和探索欲（如网暴题材、人生模拟器等）。
- **设计要求**：
  - 必须在 `page_design.md` 中清晰地标明每个选项的跳转目标（如 `-> [跳转至 页面_5A]` 或 `-> [结局 1]`）。
  - 必须在文档末尾汇总所有的【结局页面】。

### 1. 剧情设计通用规范
- **沉浸与写实**：剧情文案应简练、有代入感。在涉及特定专业领域时，**必须**合理利用 `whole_search` 搜索工具获取真实案例、新闻或网友的梗作为参考，增强写实感和亲和力。
- **卡片式阅读**：每一页的文案不宜过长，需适合在手机屏幕的单张卡片上展示。
- **视觉资产生成规范（全局一致性设定）**：在 `page_design.md` 的开头，**必须**详细设定人物形象（主人公、配角）、核心场景与环境。
  - **人物状态变化**：在高复杂度模式下，主人公在不同分支下可能会有完全不同的境遇（如阳光、抑郁、黑化、入狱等），必须详细设定不同状态下的衣着、神态、妆容，以确保后续 AI 绘图的一致性。

### 2. 分页命名与推进规范（高复杂度模式示例）
在输出 `page_design.md` 时，严格按照以下标签格式进行编写：

```markdown
# 模拟人生小程序：《[填写剧情名称]》多分支网状剧本设计

## 🎨 视觉资产生成规范（全局一致性设定）
【主人公形象】：（年龄、性别、衣着、五官。需包含不同剧情线下的状态A、状态B等）
【次要人物形象】：...
【核心场景与环境】：...

## 📖 剧本正文
【页面_cover：封面页】
[悬念文案]

【页面_0：交代背景】
[背景文案]

【页面_1：分叉点1】
[抉择情境]
[选项A]：[动作A] -> [跳转至 页面_2A]
[选项B]：[动作B] -> [跳转至 页面_2B]

... (中间发散与局部收束的网状节点) ...

【页面_5A：分叉点5（某条支线）】
[抉择情境]
[选项A]：[动作A] -> [结局 1]
[选项B]：[动作B] -> [结局 2]

## 【结局页面汇总】
【结局 1：[结局名称]】
[结局文案]

【结局 2：[结局名称]】
[结局文案]
...

【页面_epilogue：结局寄语】
（所有结局结束后统一展示的升华文案）
```

## 第二阶段：项目初始化与数据同步 (Project Setup)

1. **阅读前置规范**：
   - 必须激活并仔细阅读 `lingguang-miniapp-meta-skill`，确保遵循灵光小程序的底层开发规范。
2. **复制模板**：
   - 将当前目录下的模板代码（如 `demo_workspace_for_virtual_life`）完整复制为新的工作目录（如 `workspace`）。
   - **模板兼容性说明**：该模板**完全兼容**高复杂度发散模式！模板底层的 React 状态机和 UI 渲染逻辑并不关心剧情是线性的还是网状的，只要数据结构正确即可。
3. **同步剧情数据 (`storyData.ts`)**：
   - 将设计好的剧情树转化为 `storyData.ts` 中的数据格式。
   - **高复杂度模式注意事项（极易出错，请务必注意）**：
     - 必须确保每个包含选项的页面，其选项被正确解析为 `choices: [{ text: "...", nextPageId: "..." }]` 数组结构，**绝对不能**把选项文本混在 `text` 正文字符串中。
     - 必须确保所有结局节点（如 `ending_1`, `ending_2`）都包含 `nextPageId: "page_epilogue"`（如果需要统一寄语）或者 `isEnding: true`（如果直接结束）。
     - 强烈建议编写 Python 脚本（利用正则或 AST）来自动解析 `page_design.md` 并生成 `storyData.ts`，避免手动编写导致漏掉 `nextPageId` 或结构错误。

## 第三阶段：视觉资产生成 (Image Generation)

1. **调用工具与并行策略**：
   - 使用 `image_generator` 子智能体生成背景图片。
   - **支持并行**：最多可并行调用 **5个** `image_generator` 任务以提高效率。你可以直接并行调用，无需使用 `async_spawn_subagent__spawn_parallel` 这个工具（该工具不稳定）。切记，最多5个。
2. **图片规格与画风要求**：
   - **尺寸**：严格遵循 2:3 宽高比。
   - **画风**：画风必须高度统一（如：2D comic 画风、现代写实插画风等）。
   - **角色与场景一致性**：严格参考 `page_design.md` 中的【视觉资产生成规范】。
3. **画面细节约束（极其重要）**：
   - **绝对无文字**：生成的图片中**不要有任何文字或台词内容**。
   - **分镜限制**：单分镜。
   - **屏幕特写规则**：如果画面需要体现电脑或手机屏幕的具体内容，**不要在同一画面中画出人物**，避免透视错误。
4. **文件存放与命名**：
   - 所有生成的图片必须放置在 `workspace/assets/image/` 目录下。
   - **命名规范**：**绝对不可以是中文**。必须使用英文和数字（如 `page_cover.png`, `page_0.png`, `ending_1.png`）。

## 第四阶段：音频资产生成 (Audio Generation)

1. **旁白配音生成**：
   - 依次调用 `audio_generator` 为每个页面的文字内容制作配音（mp3格式）。
   - **支持并行**：最多可并行调用 **5个** `audio_generator` 任务。
   - **音色选择**：根据题材选择合适的音色（如 `zh_female_cancan_uranus_bigtts` 等）。
   - **文件存放与命名**：放置在 `workspace/assets/audio/` 目录下，命名必须为纯英文/数字，与页面ID对应。
2. **背景音乐 (BGM)**：
   - 提取现有的背景音乐文件（如 `bg_music.mp3`），直接复制到 `workspace/assets/audio/` 目录中。

## 第五阶段：代码适配与交互逻辑 (Code Adaptation)

1. **资源引用 (`App.tsx`)**：
   - 在主程序文件（如 `App.tsx`）中，正确 `import` 所有新生成的图片和音频文件，并构建 `bgImages` 和 `audios` 映射对象（Record）。
   - **高复杂度模式注意事项**：由于高复杂度模式下页面数量庞大（可能多达 30-40 个），手动在 `App.tsx` 中写 `import` 极易遗漏或报错（如 `Cannot find name 'audios'`）。**强烈建议编写 Python 脚本**，扫描 `assets/image` 和 `assets/audio` 目录，自动向 `App.tsx` 注入 `import` 语句和映射对象。
2. **音频引擎规范（严格遵守）**：
   - 根据灵光小程序规范，**严格禁止使用原生音频 API**（包括 `<audio>` 标签、`HTMLAudioElement`、`new Audio()` 等）。
   - 必须从 `@/lib/audio` 导入 `Howl`，并使用 `Howl` 实例来管理背景音乐和旁白语音的播放、暂停和卸载。
3. **背景音乐播放逻辑**：
   - 封面自动播放，进入正文后默认关闭（暂停），保留用户手动控制按钮。

## 第六阶段：全局约束与执行纪律 (Global Constraints)

1. **禁止调用 Developer/Evaluator**：
   - 全程**不要**调用 `developer` 和 `evaluator` 子智能体。所有的文件复制、数据替换、代码修改均由当前主控 Agent 通过 `bash` 执行 Python/Shell 脚本自主完成。
2. **路径安全**：
   - 在执行 `bash` 命令时，注意处理路径中的特殊字符，避免文件被生成到错误的临时目录中。建议使用相对路径或安全的绝对路径。
3. **连贯执行与自检**：
   - 资产生成和代码修改应一气呵成。
   - 完成后主动执行 `npm run build` 检查是否有 TypeScript 报错或资源缺失，确保代码无语法错误。

## 第七阶段：常见事故与避坑指南 (Post-Mortem & Anti-Patterns)

在过往的开发中，AI 智能体曾犯下过一些严重的执行偏差。为了避免重蹈覆辙，请在开发时**反复核对**以下避坑指南：

### 1. 首页白屏事故 (White Screen Issue)
- **错误做法**：在根节点使用 `h-full`（`height: 100%`），并遗漏 `<div id="safe-area"></div>`。这会导致在小程序环境中高度塌陷为 0，画面完全不可见。
- **正确做法**：必须使用 `h-screen`（`height: 100vh`）确保撑满屏幕，并严格保留 `<div id="safe-area"></div>` 节点。

### 2. 音频违规拦截 (Audio Violation)
- **错误做法**：手写 `<audio>` 标签或使用 `new Audio()`、`HTMLAudioElement`。
- **正确做法**：严格遵守灵光规范，必须从 `@/lib/audio` 导入 `Howl`，并使用 `Howl` 实例来管理背景音乐和旁白。

### 3. 背景音乐逻辑遗漏 (BGM Logic)
- **错误做法**：背景音乐从头响到尾，没有在进入正文时自动关闭。
- **正确做法**：封面自动播放，**进入正文后必须默认关闭（暂停）**。在代码实现上，需要在 `handleChoice` 离开封面页时主动设置 `setIsBgMusicEnabled(false)`，并保留用户手动控制按钮。

### 4. UI 细节与审美偏差 (UI Detail Deviations)
- **音量键设计**：绝对不能使用简陋的 Emoji（如 🔊/🎵）和写死的绝对定位（如 `top-4`）。必须使用 **SVG 矢量图标**，并采用动态视口定位（如 `top-[calc(15vh-25px)]` 和 `top-[calc(15vh+25px)]`），以确保在不同屏幕高度下的视觉比例完美。
- **选项按钮排版**：选项按钮的文字必须**居中对齐**（使用 Tailwind 的 `text-center justify-center`），不要使用左对齐（`text-left`）或在右侧添加多余的箭头（`→`）。
