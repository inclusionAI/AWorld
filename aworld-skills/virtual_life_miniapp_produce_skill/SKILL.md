# 虚拟人生/人生副本类灵光小程序制作指南 (Virtual Life Mini-App Production Skill)

本指南旨在为 AI 智能体提供一套标准化的工作流，用于从零开始构建“虚拟人生”、“人生副本”或“互动视觉小说”类的灵光小程序。请严格按照以下流程和规范执行。

## 第一阶段：背景调研与剧情设计 (Story Design)

本阶段的核心目标是指导策划与AI在设计剧情时，如何规范、细致地输出分页剧情推进文档（`page_design.txt`）。核心是确保输出的剧情文档具有**严格的分页结构**、**清晰的分支逻辑**，并完美契合“幻觉选择（殊途同归）”的核心机制。

### 1. 核心设计理念
- **幻觉选择 (Illusion of Choice)**：剧情必须包含 N 个关键的分叉点。无论用户在分叉点选择A还是B，剧情在给出不同的即时反馈后，都必须巧妙地收束到同一条主线上，最终走向同一个具有警示意义的结局。
- **沉浸与写实**：剧情文案应简练、有代入感。在涉及特定专业领域（如走私、网暴、金融、历史等）时，**必须**合理利用 `whole_search` 搜索工具获取真实案例、新闻或历史背景作为参考，增强写实感。
- **卡片式阅读**：每一页的文案不宜过长，需适合在手机屏幕的单张卡片上展示。

### 2. 分页命名与推进规范
在输出 `page_design.txt` 时，**必须且只能**保留核心的页面剧情设计，严格按照以下标签格式和顺序进行编写：
- **人物形象**：主人公和次要角色的人物要有统一的设计和在视觉上明显的区分，有年龄、性别、高矮、人种、衣着、五官、配饰等维度。

- **封面与结局升华**：
  - `【页面_cover：封面页】`：在剧情开始前，提供一个吸睛的封面，包含标题和引人入胜的悬念文案，引导用户点击“开始体验”。
  - `【页面_epilogue：结局寄语】`：在剧终页面之后，提供一个立意升华的页面，总结整个故事的教育意义或警示名言，并提供“重新开始”按钮。
- **背景引入**：
  - `【页面_0：交代背景】`：设定初始情境，交代主角面临的诱惑或困境。
- **分叉点结构（循环 N 次）**：
  - `【页面_X：分叉点Y】`：描述当前面临的抉择，必须包含 `[选项A]` 和 `[选项B]`。选项A/B的文字要简洁，一般不超过10字。
  - `【页面_X+1a：分叉点Y反馈A】`：玩家选择A后的即时剧情反馈。
  - `【页面_X+1b：分叉点Y反馈B】`：玩家选择B后的即时剧情反馈。
  - `【页面_X+2：剧情推进Y】`：将A和B两条支线重新收束，推进时间线或剧情发展，为下一个分叉点做铺垫。（注：最后一个分叉点后直接接剧终页面，不需要剧情推进）。
- **结局**：
  - `【页面_Z：剧终页面】`：点明最终的悲剧/惩罚/宿命结局。

当被要求设计新的剧情时，请直接按照上述【标准化输出模板】生成内容，并写入到 `page_design.txt` 中。**无需**在文件中包含复杂的UI布局描述或数值系统设定，**只保留最核心的、按页面严格划分的剧情文案**。

## 第二阶段：项目初始化与数据同步 (Project Setup)

1. **阅读前置规范**：
   - 必须仔细阅读 `lingguang-miniapp-meta-skill.md`，确保遵循灵光小程序的底层开发规范。
2. **复制模板**：
   - 将当前目录下的模板代码（如 `demo_src_for_virtual_life`）完整复制为新的工作目录（如 `src_v1`）。
3. **同步剧情数据**：
   - 将设计好的剧情树转化为代码所需的数据格式（如 `storyData.ts` 或相应的数据文件），确保页面ID和跳转逻辑完全对应。

## 第三阶段：视觉资产生成 (Image Generation)

1. **调用工具与并行策略**：
   - 使用 `image_generator` 子智能体生成背景图片。
   - **支持并行**：最多可并行调用 **5个** `image_generator` 任务以提高效率。你可以直接并行调用，无需使用async_spawn_subagent__spawn_parallel这个工具，这个工具不稳定。切记，最多5个。
2. **图片规格与画风要求**：
   - **尺寸**：严格遵循 3:4 宽高比，分辨率为 `1024x1536`。
   - **画风**：画风必须高度统一，且与剧情主题贴合（如：中国国内场景、高质量漫画风格（比如小浣熊水浒卡）等）。
   - **时代**：年代，古代（具体哪个朝代，中国的还是国外的）还是近代，现代，当代，这些要有明确的、统一的要求。
   - **角色一致性**：如果有主人公，主人公的服装、外貌特征必须在所有图片中保持一致。甚至在一开始的page_design.txt里面写好主人公以及关键人物的统一设计，然后将其坚定的贯穿于所有这些人物出现的画面中，严格确保人物形象的一致性。
3. **画面细节约束（极其重要）**：
   - **绝对无文字**：生成的图片中**不要有任何文字或台词内容**。
   - **分镜限制**：每个图片最多包含一到两个分镜，最好是单分镜。
   - **屏幕特写规则**：如果画面需要体现电脑或手机屏幕的具体内容，**不要在同一画面中画出人物**。因为同时显示人物和屏幕时，AI容易产生空间透视错误（如屏幕内容出现在设备背面）。
4. **文件存放**：
   - 所有生成的图片必须放置在 `src_v1/assets/image/` 目录下（output path 务必写相对路径），命名需与页面ID对应（如 `page_cover.png`, `page_0.png`）。

## 第四阶段：音频资产生成 (Audio Generation)

1. **旁白配音生成**：
   - 依次调用 `audio_generator` 为每个页面的文字内容制作配音（mp3格式）。
   - **支持并行**：最多可并行调用 **5个** `audio_generator` 任务。
   - **音色选择**：根据题材选择合适的音色（如无特殊要求，可使用沉稳的男声 `zh_male_liufei_uranus_bigtts` 等）。
   - **文件存放**：放置在 `src_v1/assets/audio/` 目录下（output path 务必写相对路径），命名与页面ID对应。
2. **背景音乐 (BGM)**：
   - 提取现有的背景音乐文件（如 `bg_music.mp3`），直接复制到 `src_v1/assets/audio/` 目录中。

## 第五阶段：代码适配与交互逻辑 (Code Adaptation)

1. **资源引用**：
   - 在主程序文件（如 `App.tsx`）中，正确 `import` 所有新生成的图片和音频文件，并映射到对应的页面ID字典中。
2. **背景音乐播放逻辑**：
   - **封面自动播放**：当处于封面页 (`page_cover`) 时，背景音乐默认开启并播放。
   - **其他页面默认关闭**：进入正文页面后，背景音乐默认关闭（暂停）。
   - **用户控制**：保留页面上的音乐控制按钮，允许用户在任何页面手动开启或关闭背景音乐。

## 第六阶段：全局约束与执行纪律 (Global Constraints)

1. **禁止调用 Developer**：
   - 全程**不要**调用 `developer` 子智能体。所有的文件复制、数据替换、代码修改（如正则替换、AST修改）均由当前主控 Agent 通过 `bash` 执行 Python/Shell 脚本自主完成。
2. **路径安全**：
   - 在执行 `bash` 命令时，注意处理路径中的特殊字符（如括号 `()`、空格等），避免文件被生成到错误的临时目录中。建议使用相对路径或安全的绝对路径。
3. **连贯执行**：
   - 资产生成和代码修改应一气呵成，完成后主动检查目录结构，确保所有 `.png` 和 `.mp3` 文件均已就位且代码无语法错误。
